La importancia de la firma, como se crea:
Primero vamos con el Signature del Assertion:
Signature contiene SignedInfo, SignatureValue y KeyInfo

Se supone que el Assertion Tag es signed, es decir se pasa por un SHA-256 y el SignedInfo contiene ese digest en DigestValue,
Hasta aqui todo es estandard, salvo el DigestValue que tiene el valor del assertion tag pasado por sha-256 y la etiqueta
Reference con el atributo URI que hace referencia al Assertion tag, todo lo demas no cambia.

Depues se supone que se pase por otro sha-26 el SignedInfo completo y se encripta con la private key del IdP y ese valor
v como valor del tag SignatureValue

Finalmente como sefirmo con la private key de IdP, se manda a certificado con la llave publica en el tag KeyInfo.

El tag Signature va dentro de tag Assertion.

De forma similar se crea otro tag Signature que finalmente se mete en el tag raiz Response, 
esto pasa porque en okta declaramos sha-256 como el digest, RSA para firmar y encriptar y que estos dos
pedazos , a respueta y el assertion, iban a regresar firmados.

Depues debemos usar la inversa, el valor de SignatureValue es la la firma encriptada, asi que
podemos desencriptar con la public key del certificado, eso nos dara un digest, el segundo Digest
creado a partir del SignedInfo en el paso de firmar.

Despues, crear el digest del SignedInfo y verificar que los dos digest correspondan.

Eso es la primera parte, despues falta validar el xml original, si estamos verificand AssertionTag,
ese seria el XML original pero removiendo el tag Signature, nuevamente se creria un digest de ese xml
y ese valor debe ser el mismo que el valor de la etiqueta DigestValue, todo esto es la inversa.
